Guidelines

Vous connaissez maintenant le principe du peer-correcting.
Si ce n'est pas encore le cas, retrouvez les directives sur l'état d'esprit du peer-correcting dans le document dédié sur l'e-learning.

Vérifiez d'abord les éléments suivants :

- Il y a bien un rendu (dans le dépôt git)
- Fichier auteur valide
- Le Makefile est présent et compile bien l'exécutable philo
- Pas de faute de norme, la Norminette faisant foi
- Pas de triche (fonctions non autorisées, les étudiants doivent pouvoir expliquer leur code, ...)

Si un élément n'est pas conforme au sujet, la notation s'arrête la. Vous êtes encouragés à continuer de débattre du projet, mais le barème n'est pas appliqué.

Préliminaires

Exigences du sujet

Si l'un des points suivant n'est pas respecté, le projet est considéré hors sujet et la soutenance s'arrête immédiatement. Vous pouvez néanmoins continuer à débattre du sujet, sans appliquer le barême.

- Les #defines suivants doivent être présents dans le projet: MAX_LIFE, THINK_T, EAT_T, REST_T, TIMEOUT.
- Plusieurs philosophes doivent pouvoir être dans l'état "manger" au même moment.
- Les philosophes ne doivent pas manger dans un ordre arbitraire si l'on execute plusieurs fois le programme avec les mêmes paramètres.
- 7 philosophes ou plus doivent pouvoir être à table ensemble.
- Le nombre de baguettes doit TOUJOURS correspondre EXACTEMENT au nombre de philosophes. Ni plus, ni moins (sauf bonus explicitement démontré).
- Les philosophes passent bien par les trois états: réfléchit / mange / se repose (think / eat / rest).
- Les philosophes qui ne mangent pas perdent 1 point de vie par SECONDE de jeûne.

Général

Comportement des philosophes

Vérifiez que les philosophes peuvent mourir de faim dans de mauvaises conditions (testez un cas concret).

Interface graphique

Il y a une interface graphique fonctionelle (mlx, SDL, ncurses, OpenGL, ...) ?

Threading

Multi-thread

Il y a plusieurs threads qui tournent en simultané ?

1 thread par philosophe

Vérifiez qu'il y a un et un seul thread par philosophe (sauf threads graphiques éventuels).

Création/destruction des threads

Les threads sont correctement crées et détruits aux bons moments (pthread_create et pthread_detach) ?

Mutexes & ressources

Ce qui compte ici c'est le partage des ressources, il doit être équitable pour valider la question.

1 mutex par baguette

Chaque baguette doit posséder son propre mutex.

Usage des mutexes

Les mutexes empêchent correctement les threads d'accéder aux baguettes en cours d'utilisation ?

lock et trylock

On a une utilisation intelligente de pthread_mutex_lock et pthread_mutex_trylock ? Les étudiants doivent justifier dans quels cas ils utilisent l'un et l'autre.

Pas de boucle

Aucune boucle ne doit être utilisée pour attendre le déblocage d'un mutex.

=> A proscrire :
while (pthread_mutex_trylock(...) != 0)

Si une boucle est utilisée, laissez Non.

Stabilité

Vous allez tester la stabilité du programme. Pour ce test il y aura 7 philosophes autour de la table, vous laisserez les valeurs par défaut choisies par l'étudiant pour chacun des #define.

Lancez le programme.

- Si une nette tendance à la baisse se dessine dans les points de vie d'un ou plusieurs philosophes, le programme n'est pas assez stable.
- Si les philosophes ne s'affament pas petit à petit, c'est à dire que le programme pourrait continuer à s'exécuter indéfiniment, le programme est stable.

Performances

Pour les trois tests suivants, le nombre philosophes à table sera de 7 et le TIMEOUT sera mis à 60 secondes. Si passé ce délai personne n'est mort, cochez Oui. Pour être sûr, vous effectuerez deux fois chaque test (vous pouvez le faire en parallèle pour aller plus vite). Le meilleur résultat comptera.

Test 1

Testez avec MAX_LIFE = 10, EAT_T = 1 seconde, REST_T = 1 seconde, THINK_T = 1.

Test 2

Testez avec MAX_LIFE = 7, EAT_T = 1 seconde, REST_T = 1 seconde, THINK_T = 1.

Test 3

Testez avec MAX_LIFE = 5, EAT_T = 1 seconde, REST_T = 1 seconde, THINK_T = 1.
