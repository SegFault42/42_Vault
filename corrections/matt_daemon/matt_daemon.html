Introduction


Nous vous demandons pour le bon déroulement de cette évaluation de
respecter les règles suivantes :

- Restez courtois, polis, respectueux et constructifs en toutes
situations lors de cet échange. Le lien de confiance entre la
communauté 42 et vous en dépend.

- Mettez en évidence auprès de la personne (ou du groupe) notée les
dysfonctionnements éventuels du travail rendu, et prenez le temps
d'en discuter et d'en débattre.

- Acceptez qu'il puisse y avoir parfois des différences
d’interprétation sur les demandes du sujet ou l’étendue des
fonctionnalités. Restez ouvert d'esprit face à la vision de
l'autre (a-t-il ou elle raison ou tort ?), et notez le plus
honnêtement possible. La pédagogie de 42 n'a de sens que si
la peer-évaluation est faite sérieusement.

Guidelines


- Vous ne devez évaluer que ce qui se trouve sur le depôt GiT de rendu
de l'étudiant(e) ou du groupe.

- Prenez soin de vérifier que le depôt GiT est bien celui
correspondant a l'étudiant(e) ou au groupe, et au projet.

- Verifiez méticuleusement qu'aucun alias malicieux n'a été utilisé
pour vous induire en erreur et vous faire évaluer autre chose que
le contenu du dépot officiel.

- Tout script sensé faciliter l'évaluation fourni par l'un des deux
partis doit être rigoureusement vérifié par l'autre parti pour
éviter des mauvaises surprises.

- Si l'étudiant(e) correcteur/correctrice n'a pas encore fait ce
projet, il est obligatoire pour cet(te) etudiant(e) de lire le
sujet en entier avant de commencer cette soutenance.

- Utilisez les flags disponibles sur ce barème pour signaler un
rendu vide, non fonctionnel, une faute de norme, un cas de triche,
etc. Dans ce cas, l'évaluation est terminée et la note finale est
0 (ou -42 dans le cas special de la triche). Toutefois, hors cas
de triche, vous etes encouragés a continuer d'échanger autour du
travail éffectué (ou non effectué justement) pour identifier les
problemes ayant entrainé cette situation et les éviter pour le
prochain rendu.

- L'évaluation doit se passer dans une machine virtuelle sous linux
ayant un noyau > 3.14 en disposant des droits root.

Préliminaires.

Préliminaires

Avant de commencer la soutenance, veuillez vérifier les points suivants :

- le projet est en C++,
- le projet utilise bien les fonctions fork, signal ainsi que chdir,
- le flag LOCK_SH n'est pas visible dans le projet,
- les flags LOCK_EX ainsi que LOCK_UN sont présent dans le projet,
- une classe Tintin_reporter est bien présente,
- les classes ont bien la forme de Coplien,
- un Makefile contenant les règles usuelles est présent.

Vérifiez également que le projet n'utilise pas de bibliothèque qui nuit à l'intérêt pédagogique du projet. Si c'est le cas, cochez "Cheat" en haut du barème.

Si un seul de ces points est invalide, la correction s'arrête.

Matt_daemon use

Lancement daemon

Avant de lancer le programme, lancez la commande suivante : `rm -f /var/lock/matt_daemon.lock ; rm -rf /var/log/matt_daemon/`.

Lancez le programme en utilisateur root.

La commande "netstat -tulpn" vous permettra de vérifier que le port 4242 est bien en mode LISTEN.

Les fichiers /var/lock/matt_daemon.lock et /var/log/matt_daemon/matt_daemon.log sont bien disponibles.

Fermeture du daemon

Recherchez le PID du daemon via la commande `ps aux | grep matt_daemon`, puis faites un kill -15 (PID du daemon).

Le fichier /var/lock/daemon.lock ne doit plus être disponible.

Si une condition n\'est pas respectée, alors la correction s\'arrête ici.

Matt_daemon usage

Matt_daemon utilisation

Avant de lancer le programme, lancez la commande suivante : `rm -f /var/lock/matt_daemon.lock ; rm -rf /var/log/matt_daemon/`.

Lancez le programme avec les droits root.

Utilisez la commande netcat pour vous connecter au daemon ("nc localhost 4242").

Écrivez un mot dans l'invite de commande, puis envoyez (ne pas écrire quit).

Terminez la session en tapant "quit" dans l\'invite de commande.

Le programme doit se terminer correctement et le fichier /var/lock/matt_daemon.lock ne doit plus être présent.

Matt_daemon journalisation

Matt_daemon journalisation simple

Le fichier /var/log/matt_daemon/matt_daemon.log est bien présent.
On distingue assez clairement les informations.
Le lancement et la fermeture du daemon sont lisibles clairement.
Les instructions tapées dans l\'invite de commande sont bien lisibles.
Le timestamp est bien dans le format demandé dans le sujet.
Un message indique l'appel à la fermeture du daemon via la commande quit.

Matt_daemon journalisation avancée

Relancez le daemon puis killez-le via le signal de votre choix.
Si un signal n\'est pas correctement géré, la personne corrigée doit pouvoir le justifier.
Le daemon se referme bien correctement avec la destruction du fichier /var/lock/matt_daemon.lock.
Le signal est bien visible dans notre fichier de log /var/log/matt_daemon/matt_daemon.log.

Matt_daemon gestion erreur

Matt_daemon error basique

Avant de lancer le programme, lancez la commande suivante `rm -f /var/lock/matt_daemon.lock ; rm -rf /var/log/matt_daemon/`.
Tentez de lancer le daemon en root deux fois.
Un message d\'erreur explicite s\'affiche et ne lance pas le second daemon.
Un message propre est visble dans le fichier /var/log/matt_daemon/matt_daemon.log, avec une explication du problème.

Matt_daemon error avancée

Avant de lancer le programme, lancez la commande suivante `rm -f /var/lock/matt_daemon.lock ; rm -rf /var/log/matt_daemon/`.
Lancez le daemon en root.
Le correcteur est libre de tenter de faire planter le programme (sans SIGKILL).
En cas d\'arrêt imprévu, le daemon doit quitter proprement et un message doit être enregistré dans le fichier de journalisation.

Matt_daemon socket

Matt_daemon 1

Lancez le daemon en root.
Tentez de connecter plus de 3 clients en simultané sur le daemon.
Un message d\'erreur doit être lisible pour indiquer le problème.

Matt_daemon 2

Lancez le daemon en root.
Tentez de connecter 3 clients en simultané sur le daemon.
Quittez un client puis le relancer.
Le client doit pouvoir se connecter simplement, sans erreur.
Fermez alors tous les clients ainsi que le daemon proprement.
