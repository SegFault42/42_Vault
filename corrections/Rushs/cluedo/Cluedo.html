Introduction

Préliminaires

Tests préliminaires

Vérifiez d'abord les points suivants :

- vous n'évaluez que le travail se trouvant sur le dépôt obtenu par git clone
- il y a bien un fichier auteur valide
- les fichiers de la partie obligatoire (lorsque les exercices sont rendus) s'appellent step1.pl, step2.pl et step3.pl

Pour ce rush, il est tout à fait concevable que les étudiants se soient inspirés de code existant pour réussir les énigmes. On ne vous demande pas, après tout, de ré-inventer le Prolog en un weekend.

Il est en revanche inadmissible de rendre un code que l'on ne comprend pas et que l'on ne peut pas reproduire. Pour cette raison, pour chaque énigme de ce rush, les corrigés devront prendre le temps d'expliquer leur code au correcteur. S'ils en sont incapables, ils ont 0.

Step 1

Interroger la base de connaissances

Le principe de la programmation logique est la déclaration de connaissances, et l'interrogation de cette base de connaissances par un moteur d'inférence.
Vous allez interroger la base de connaissances du fichier step1.pl. Ouvrez XPG, le fichier via menu File -> Open, et dans la fenêtre du fichier, menu Tools -> Evaluate Query.

Les résultats doivent être strictiment identiques à ceux du barême pour valider la question. Pour chercher toutes les solutions d'une requête, sélectionnez Evaluate : All.

Utilisation de règles

Examinez la base de connaissances. Si les corrigés ont été malins, certains prédicats auront été implémentés sous forme de règles. Les règles sont une forme composée permettant de déduire un fait à partir d'autres faits.

Typiquement, pour décrire le prédicat bellemere_de(X, Y) on devrait avoir un prédicat de la forme :

bellemere_de(X, Y) :- femme(X), mere_de(X, Z), epoux(Z, Y).

Vous ne devez pas avoir simplement une suite de faits de la forme suivante :
bellemere_de(anne, lucie).

Ne validez la question que si les étudiants ont utilisé des règles.

Résolution des énigmes

Step 2 : l'énigme d'Einstein

Vérifiez d'abord que le programme de l'étudiant fournit une seule et unique solution au problème. Pour cela, chargez le fichier step2.pl et lancez la requête en sélectionnant Evaluate : All.

Si le programme est fonctionnel, et que l'on découvre effectivement que c'est l'allemand qui a le poisson, c'est le moment d'examiner le code.

Les corrigés doivent commenter et expliquer leur code. Ils doivent pouvoir expliquer, en termes simples, la logique du programme. Si ce n'est pas le cas, la question est invalide.
Ce point est important, il n'y a aucun intérêt pédagogique à rendre un code que l'on ne comprend pas.

Step 3 : l'énigme de la chèvre, du loup et du chou

Vérifiez d'abord que le programme de l'étudiant fournit une seule et unique solution au problème. Pour cela, chargez le fichier step3.pl et lancez la requête de résolution en sélectionnant Evaluate : All..

Le programme doit afficher les états successifs des rives, ou les traversées successives, ou tout autre mécanisme permettant d'expliciter les différentes étapes de la résolution de l'énigme.

Si le programme est fonctionnel, et qu'il propose une solution exacte (une seule, comme demandé dans le sujet), c'est le moment d'examiner le code.

Les corrigés doivent commenter et expliquer leur code. Ils doivent pouvoir expliquer, en termes simples, la logique du programme. Si ce n'est pas le cas, la question est invalide.
Ce point est important, il n'y a aucun intérêt pédagogique à rendre un code que l'on ne comprend pas.
