Preliminaires

C'est parti. Ceci est le bareme du premier jour de PHP. Il y a tout plein d'exos, restez pas trois plombes sur chaque exo, sinon vous ne vous en sortirez pas. Realisez des tests simples rapides et efficaces. Normalement vous avez teste plein de trucs pour vos propres exos, donc re-testez les memes. ATTENTION: ce bareme ne va pas vous dire quels tests effectuer. Oui, certains feront peut-etre plus(+) de tests bizarres, et donc vautrer plus souvent les programmes (et arreter l'evaluation). Solution: multipliez les soutenances pour ecreter le phenomene. Edit: ce n'est a priori pas clair pour tout le monde. Des qu'un exo est faux, la notation s'arrete, l'evaluation s'arrete, on ne note pas les exos suivants. Vous pouvez toujours en discuter entre vous, mais en tout cas, on note rien dans l'intra. C'est egalement note dans les consignes du sujet, au point numero 7.

De base

Classiquement:
- soutenance en presentiel
- git clone quand tout le monde est la
- pas de rendu : 0
- sauf si c'est precise, une erreur php temoigne d'une chose mal geree dans le code. Donc l'exo vaut 0 et l'evaluation s'arrete. (c'est comme un message d'erreur a la compil ou un segfault en C).

Les faciles

Les premiers exos qui sont plutot simples..

HW

Si le prog affiche un Hello World, c'est bon.
Sinon la soutenance s'arrete.

mlX

Faites un ls pour verifier la taille du script php, celle-ci doit faire moins de 100 caractères (tout le fichier, donc tout compris).
Verifier egalement le nb de X, notament qu'il n'y en ai pas 999 ou 1001.
Ca passe, ou ca casse.
(rappel: si ca casse, fin de l'evaluation, on regarde pas les exos suivants).

Au divin

Le prog doit demander un nb en ligne de commande et dire si il est pair ou impair.
- le resultat doit etre ok
- l'affichage des messages doit etre exact
- les non-nombres doivent etre détectés
- tester a minima les exemples du sujet, plus c'est mieux.
- pas de test avec les nb flottants.

ft_split

Refaites la manip du sujet, verifiez que cela fonctionne normalement.
Rajoutez des tests.

aff_param

Ca marche ?
Je vous laisse tester.

Un peu moins simple

(mais tout juste alors)

epur_str

- verifiez que les espaces sont bien reduits a 1
- verifiez que les eventuels espaces en debut et fin de chaine sont aussi enleves.

SSAP

Donc plusieurs parametres, dont certains avec des espaces un peu partout.
L'ensemble des mots sont affiches, dans l'ordre de tri ascii.

rostring

Comme epur_str mais en plus le premier mot se retrouve en dernier lors de l'affichage.
A vos tests !

ft_is_sort

reprenez l'exemple du sujet, a minima, pour tester l'utilisation de la fonction ft_is_sort.

C'est un strcmp / ordre ascii qui est en vigueur pour l'ordre des elements.

L'ordre ascendant et l'ordre descendant sont acceptes.
( "a","b","c" ok et "c","b","a" aussi).

plus difficile

ca se complique !

ssap2

Pour rappel, les differents parametres sont decoupes en mots (uniquement separes par des espaces).
Les differents mots sont tries selon l'ordre particulier decrit dans le sujet.
La comparaison est faite char par char.
=> az < a4 < a# < Cb < 42 < 5v < _toto < _TOT0

do_op

Alors allez-y.
Testez donc do_op.

Pas de gestion d'erreur a faire, hormis le nb de parametre a controler. Donc quand vous testez 3 parametres, vous mettez 3 parametres corrects, contenant eventuellement des espaces.

Pas de division par 0.

do_op_2 : la version ++

Cette fois-ci, un seul parametre. Si il y en a plus ou pas assez, on retrouve le message d'erreur de l'exo precedent.
Verifiez que quelque chose qui ne ressemble pas a un calcul fasse bien syntax error.
Si vous testez le dernier exemple (celui avec rm -rf entre back-quotes), faites-le sur le compte de l'etudiant soutenu. Pas le votre. Enfin, c'est comme vous voulez.

On ne teste pas la division par 0.
