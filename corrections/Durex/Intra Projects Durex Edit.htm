    Introduction
    

    


    Nous vous demandons pour le bon déroulement de cette évaluation de

    respecter les règles suivantes :

    

    - Restez courtois, polis, respectueux et constructifs en toutes

      situations lors de cet échange. Le lien de confiance entre la

      communauté 42 et vous en dépend.

    

    - Mettez en évidence auprès de la personne (ou du groupe) notée les

      dysfonctionnements éventuels du travail rendu, et prenez le temps

      d'en discuter et d'en débattre.

    

    - Acceptez qu'il puisse y avoir parfois des différences

      d’interprétation sur les demandes du sujet ou l’étendue des

      fonctionnalités. Restez ouvert d'esprit face à la vision de

      l'autre (a-t-il ou elle raison ou tort ?), et notez le plus

      honnêtement possible. La pédagogie de 42 n'a de sens que si

      la peer-évaluation est faite sérieusement.

    

    

    

    

    Guidelines
    

    


    - Vous ne devez évaluer que ce qui se trouve sur le depôt GiT de rendu

      de l'étudiant(e) ou du groupe.

    

    - Prenez soin de vérifier que le depôt GiT est bien celui

      correspondant a l'étudiant(e) ou au groupe, et au projet.

    

    - Verifiez méticuleusement qu'aucun alias malicieux n'a été utilisé

      pour vous induire en erreur et vous faire évaluer autre chose que

      le contenu du dépot officiel.

    

    - Tout script sensé faciliter l'évaluation fourni par l'un des deux

      partis doit être rigoureusement vérifié par l'autre parti pour

      éviter des mauvaises surprises.

    

    - Si l'étudiant(e) correcteur/correctrice n'a pas encore fait ce

      projet, il est obligatoire pour cet(te) etudiant(e) de lire le

      sujet en entier avant de commencer cette soutenance.

    

    - Utilisez les flags disponibles sur ce barème pour signaler un

      rendu vide, non fonctionnel, une faute de norme, un cas de triche,

      etc. Dans ce cas, l'évaluation est terminée et la note finale est

      0 (ou -42 dans le cas special de la triche). Toutefois, hors cas

      de triche, vous etes encouragés a continuer d'échanger autour du

      travail éffectué (ou non effectué justement) pour identifier les

      problemes ayant entrainé cette situation et les éviter pour le

      prochain rendu.

    Durex source
    

    Préliminaires.
    

    Préliminaires
    

    


    Avant de commencer la soutenance, veuillez vérifier les points suivants :

    - L'évaluation doit se passer dans une machine virtuelle sous le système

      d'exploitation choisi par le corrigé en disposant des droits root.

    - Le projet est en C ou en assembleur,

    - Le projet utilise bien les fonctions fork ainsi que chdir,

    - Un Makefile contenant les règles usuelles est présent.

    

    Vérifiez également que le projet n'utilise pas de bibliothèques qui nuit

    à l'intérêt pédagogique du projet, comprendre que la(les) bibliothèque(s)

    ne font pas le café à elle(s)-seule(s).

    

    Si un seul de ces points est invalide, la correction s'arrête.

    

    

    

    

    

    





    

    

    

    

    Durex utilisation
    

    

    Lancement trojan
    

    


    Le corrigé doit pouvoir montrer que le port 4242 est pas en mode LISTEN.

    

    Lancez le programme en utilisateur root.

    Le login du corrigé doit être visible sur la sortie standard.

    Le programme se termine correctement sans erreurs.

    

    Le corrigé doit pouvoir montrer que le port 4242 est maintenant bien en mode LISTEN.

    Aucune erreur ne doit être visible.

    

    Si une condition n'est pas respectée, alors la correction s'arrête ici.

    

    

    

    

    

    





    

    

    Tentative utilisation trojan
    

    


    Tentez de vous connecter via la commande "netcat  4242" sur la machine

    à distance.

    Un prompt doit alors demander un password.

    

    En cas de mauvais password, le prompt doit redemander le password.

    Lorsque le password entré est correct, un nouveau prompt est alors ouvert.

    

    Vérifiez dans le code source que le mot de passe en question n'est pas codé en clair.

    

    Si une de ces conditions n'est pas respectée, alors la correction s'arrête ici.

    

    

    

    

    

    





    

    

    Trojan utilisation
    

    


    Maintenant, relancer la machine virtuelle et retenter de se connecter sur

    cette machine par la commande "netcat  4242".

    

    Lorsque le password entré est correct, un nouveau prompt est alors ouvert.

    Lancer la commande pour ouvrir le second shell en disposant des droits root.

    

    Enfin, à la fermeture du shell reverse, la connexion se ferme correctement.

    

    Si une condition n'est pas respectée, alors la correction s'arrête ici.

    

    

    

    

    

    





    

    

    

    

    Trojan tentative erreur
    

    

    Trojan basique
    

    
Tentez de connecter plus de 3 clients en simultané sur le daemon.

    Un message d'erreur doit être lisible coté client uniquement pour indiquer le problème

    

    En dehors de ce prompt spécifique, à aucun moment le trojan ne doit afficher le moindre message d'erreur sur l'ordinateur infecté (sortie ou log).

    

    Si une condition n'est pas respectée, alors la correction s'arrête ici.

    

    

    

    

    

    





    

    

    Trojan avancé
    

    


    Le correcteur est libre de tenter de faire planter le trojan.

    En cas d'arrêt imprévu, le daemon doit quitter proprement et aucun message doit être visible sur l'ordinateur de la personne infécté.

    

    Si une condition n'est pas respectée, alors la correction s'arrête ici.

    

    

    

    

    

    





    

    

    

    

    Trojan implémentation
    

    

    Implémentation
    

    


    La méthode d'implémentation utilisé semble parfaitement adaptée aux besoin de l'utilisation.

    La note minimale est accordée au rendu basique avec très peu d'explication.

    

    L'utilisation d'un quine et l'explication de son implémentation va permettre ici d'avoir la note maximale.

    

    En cas de moindre soucis dans l'explication, la correction s'arrête.

    

    

    

    

    

    Rate it from 0 (failed) through 5 (excellent)
    

    

    

    

    

    

    Trojan bonus
    

    Les bonus ne doivent être évalués que si et seulement si la partie
obligatoire est PARFAITE. Par PARFAITE, on entend bien évidemment
qu'elle est entièrement réalisée, qu'il n'est pas possible de
mettre son comportement en défaut, même en cas d'erreur, aussi
vicieuse soit-elle, de mauvaise utilisation, etc. Concrètement,
cela signifie que si la partie obligatoire n'a pas obtenu TOUS les
points pendant cette soutenance, les bonus doivent être
intégralement IGNORÉS.

    

    Bonus faits pour ce projet
    

    


    Les bonus doivent être un minimum utiles, et bien implémentés.

    La notation est à la discrétion du correcteur.

    

    Voici des exemples de bonus recevables:

    

    - Fonctions utilitaires

    - Utilisation packing

    - Système d'authentification avancé

    - ...

    

    

    

    

    

    Rate it from 0 (failed) through 5 (excellent)
